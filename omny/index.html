1. Introduction

Omny Link is an open source CRM. Unlike many such tools it is built specifically with flexibility in mind because every customer needs to analyse their customer data and react to events as the occur (or even try to anticipate them).

This flexibility is delivered because:

  • It is built API-first: First and foremost every service of the CRM is defined as a set of micro-services that can be invoked directly by their REST API. Our user interface is just one example of doing this.

  • It embeds a full BPMN 2 process engine: Pair this industry standard for defining workflow with Omny REST services for highly agile and yet still uniquely tailored solutions.

  • It also embeds a DMN decision engine: The decision engine allows formal description of how your business makes decisions and the ability to re-configure them as ciscumstances change.

This documentation is structured around a set of user stories so you can quickly find the way that Omny is able to satisfy problems you already recognise.

2. Account Management

2.1. A.1 Create Tenancy

2.1.1. Actor: Omny Staff

2.1.2. User story

After preliminary discussions, proposal and demo the client is ready to make a commitment. This process supplies the necessary information to configure Omny for the new client (a.k.a create their tenancy).

We aim to reach this commitment point 'quickly' and as such it excludes custom or roadmap features.

2.1.3. User outcome

We (Omny) can tell the client an estimated date that the system will be available. Depending on the user stories required follow-up questionaires may be issued. Content of these questionnaires is described in the relevant stories.

For the time being this provisioning process is manual but over time we will seek to automate parts.

2.1.4. Implementation pre-requisites

None.

2.1.5. Implementation

  1. Create tenant record

    Currently assigned to developer and consists of database insert.

  2. Create tenant JSON

    Currently assigned to developer and consists of creating a tenant JSON file embedding the customisations. The ACME configuration is a good example to copy and paste.

    This file needs to be served from a web server somewhere. Assuming we host the client website then by convention is is stored within the WordPress theme for example:

    http://acme.com/wp-content/themes/acme/acme-tenant.json

    In any case the URL must be registered in the tenant record (above).

  3. Create web customisations

    Currently assigned to developer group. Web customisations refers to:

    • custom CSS over and above that provided by the simple color palette.

    • custom HTML layouts for custom fields (which by default are laid out in two columns)

  4. Verify tenancy

    This task is assigned to whoever raised the original ticket. The purpose is to verify that the configured tenancy behaves as expected.

    On completing this task the user’s administrator will be sent credentials to access the system.

  5. Create Admin record

    Currently assigned to developer. The following tasks are undertaken for the tenant’s admin:

    • Create contact

    • Create user, assigning groups User and tenant_Admin

    • Set a password for the user

    • Send email to the user

3. Contact Management

3.1. C.1 Enter Lead - offline enquiry

3.1.1. Actor: Client Staff

3.1.2. User story

The user identifies (or is contacted by) an individual who may or may not be associtaed with an organisation and who is potentially interested in the product or service offered by the tenant. The user enters the lead into the Contact page.

3.1.3. User outcome

A new contact and, optionally, account recorded in the CRM with some arbitrary set of standard and custom fields filled out.

3.1.4. Implementation pre-requisites

  • Create tenancy, inc. ‘Bot’

  • Register client users

3.1.5. Implementation

  • Configure tenant with:

  • En/Disable features

  • Contact fields

  • Enquiry processes

  • Check Tenant status page

4. Work Management

4.1. W.1 View tasks (embedded)

4.1.1. Actor: Client Staff

4.1.2. User story

At any given time there are many things happening across different aspects of a small business. To keep on top of all these things the user wants a single place to go to see the work to be done.

How people manage such work is quite a personal choice as everyone has their own habits and preferences, which is why there are so many tools on the market. Omny’s take is that we should be able to deliver the tasks to the tool of your choice so that the user and for you to use that familiar interface. Therefore we should take care that the existing embedded task view is not over-sold.
